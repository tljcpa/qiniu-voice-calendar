# 语音方案选型

> 为什么本项目用 Azure Speech，而不是浏览器自带的 Web Speech API。这是作品的核心差异化（亮点 1）。

## 结论先行

本项目用 **Azure Speech（浏览器端 JS SDK，后端签发短时 token）** 做中文 ASR 与 TTS，
而非大多数参赛者会图省事选用的浏览器 Web Speech API。

## 三个候选

| 方案 | 中文识别 | 浏览器兼容 | 成本 | TTS |
|---|---|---|---|---|
| A. Azure Speech | 工业级，明显更准 | 自带 SDK，跨浏览器一致 | 已开 F0 免费 50万字/月 | 支持，自然神经语音 |
| B. Web Speech API | 中文识别率不能看 | Safari/部分国产浏览器不支持 | 免费 | 部分支持，机械音 |
| C. 阿里云/七牛云 ASR | 较好 | 需自行集成 | 需额外开户配置 | 需另接 |

**选 A**。理由：

1. **准确率是"准确顺畅"的命门**。题目要求"准确、顺畅地"完成语音操作。Web Speech API 的中文识别在嘈杂环境、口语化表达下错误率高，直接拖垮整条 NLU 链路——识别错了，再好的意图解析也白搭。
2. **兼容性**。Web Speech API 在 Safari、部分国产浏览器上缺失或行为不一致，移动端不可控。Azure SDK 在各浏览器表现一致。
3. **闭环需要 TTS**。Azure 提供自然神经语音（zh-CN-XiaoxiaoNeural），能做"系统语音回应"的对话闭环（亮点 2）；Web Speech 的 TTS 机械、音色差。
4. **成本可控**。F0 免费层 50 万字符/月，远超 demo 需要；C 方案在已有 Azure 配额下属重复投入。

## 关键工程设计：token 而非 key

浏览器要直连 Azure，但**订阅 key 绝不能下发前端**（进了前端 JS 就等于公开，会被盗刷配额）。

做法（见 [复盘 D-02](复盘.md)）：

```
浏览器 ──请求──> 后端 /api/speech/token ──用 key 换──> Azure 短时令牌(~10min)
浏览器 ──拿令牌直连──> Azure ASR/TTS
```

- 后端用 key 调 Azure issueToken 端点换取短时令牌，缓存 9 分钟复用（降延迟、省配额）。
- 令牌即使泄露也很快失效、且不可反推 key。这是 Azure 官方推荐的浏览器集成方式。
- 后端 `speech.py` 同时保留服务端 ASR/TTS 能力，作命令行验证与浏览器不可用时的降级。

## 为什么 ASR/TTS 放浏览器而非后端中转

浏览器用 SDK 直连 Azure，省去"音频上传后端再转发"的一跳：

- **延迟更低**，能实时回显"识别中"的文字（流式体验）。
- 后端不必处理音频流，职责更轻（只做 NLU 与日历逻辑）。

代价是后端拿不到原始音频（只拿文本）——对本场景无影响。若未来需要服务端审计/录音留存，已预留 `/ws/asr` 后端中转路径。

## 可降级

前端在 Azure 不可用（token 接口 503）时，可回退到浏览器原生 Web Speech API 作保底——
代价几乎为零，但默认走 Azure 以保证识别质量。

## 答辩可展示的对比

demo 时可对同一句中文日历指令（如"下周三下午三点跟客户对接"），对比 Web Speech API 与 Azure Speech 的识别结果，直观体现"常规方案识别错、我们识别对"。
