# 中文自然语言时间解析

> 把"下周三下午三点""每周一三五早上九点"这类口语时间表达，精确归一化为 ISO8601。这是作品的深度亮点（亮点 3），也是大多数参赛者做不到的难点。

## 为什么这是难点

用户说话从不用标准日期格式。一个语音日历要"能用"，必须听懂：

- 相对天：今天 / 明天 / 后天 / 大后天
- 周内某天 + 周修饰：这周三 / 下周三 / 下下周二 / 上周五 / 周日
- 就近某天：周六（今天周五 → 本周六；若已过 → 下周）
- 年月日：六月十八号 / 5月31号 / 明年一月一号
- 相对偏移：三天后 / 一周后
- 时刻：上午/下午/晚上/中午/凌晨 + N点 / N点半 / N点一刻 / N点三刻 / N点N分
- 循环：每周一三五 / 下周一三五（展开成多个时间点）

## 关键决策：规则层为主，不是 dateparser 为主

项目交接文档原计划"dateparser 解析 + 规则补丁"。**实测推翻了这个顺序**（见 [复盘 L-04](复盘.md)）：

```
dateparser 1.4 + languages=['zh'] 实测：
  "明天下午三点" -> None
  "三天后"       -> None
  "五月二十号"   -> None
  "下周三"       -> None
  "每周一三五"   -> None
  "周一"         -> 2026-06-01 00:00（仅这个勉强，且丢了时刻）
```

dateparser 的中文支持只覆盖极少标准格式，相对/口语表达基本不认。
据此改为**自写确定性规则层为主力**，dateparser 仅作边角兜底，LLM ISO 转换作最后兜底。

### 三层兜底（见 [复盘 D-04](复盘.md)）

1. **规则层（主力）**：确定性解析上述全部模式。规则确定、可测、零延迟、零成本；循环表达必须确定性展开，LLM 算术不可靠。
2. **dateparser（边角兜底）**：规则未命中的标准数字日期等。
3. **LLM ISO（最后兜底，可选）**：农历、节假日（"国庆假期前一天""农历八月十五"）等规则难穷举的表达，标低置信度。

## 职责分离

- **意图解析（LLM）只抽取原始时间短语** `time_expr`（如"下周一三五上午九点"），**不做时间换算**。
- **时间解析器（规则）**负责把 `time_expr` 归一化为具体 ISO8601。

这样 LLM 不碰它不擅长的时间算术，规则不碰它不擅长的语义抽取，各司其职。

## 实现要点与踩过的坑

- **中文数字**：支持 十/二十/二十三/十五 等两位组合 + 阿拉伯数字。
- **星期对齐**：周一=0…周日=6，与 `datetime.weekday()` 对齐；周修饰这/本=本周、下=+1、下下=+2、上=-1。
- **就近语义**：裸"周六"取今天及以后最近一次；已过则滚下周。
- **坑 1**：相对天匹配要按 key 长度降序，否则"后天"会被当成"大后天"的子串先命中 → 偏移错。
- **坑 2**："三刻"= 45 分，不能让分钟正则把"三刻"的"三"当数字、"刻"当一刻 → 改为 N刻 = N×15。
- **未来优先**：无年份的"5月1号"若已过，滚到明年（符合"添加未来日程"的直觉）。

## 覆盖度

`backend/tests/test_time_parser.py` 用固定参考日 2026-05-29（周五）覆盖 **30+ 种表达式**，全部通过、结果可复现。

暂不支持：农历、法定节假日名（"国庆假期"）——留 LLM 兜底层与 future work。

## 循环展开示例

```
"每周一三五早上九点"（参考日 2026-05-29 周五）
  -> 2026-05-29T09:00（本周五）
     2026-06-01T09:00（下周一）
     2026-06-03T09:00（下周三）
     2026-06-05T09:00（下周五）... 展开 4 周
  is_recurring = true
```
