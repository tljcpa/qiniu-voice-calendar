## 1. 问题陈述（0:00–0:30）

口播要点：
> "日历的图形界面其实反人类——开车、做饭、走路时手不空，想加个日程很麻烦；想知道'明天有啥安排'，扫屏幕远不如一句话问出来快；老人和视障用户对图形日历更不友好。
> 但语音日历不是'加个录音按钮'那么简单：要**识别得准**、**听得懂人话**、还要**能语音回应**。我们这三点都做了。"

## 2. 语音方案选型（0:30–1:30）

口播要点：
> "大部分方案会图省事用浏览器自带的 Web Speech API，但它中文识别率不能看、Safari 和部分国产浏览器还不支持。
> 我们用 Azure Speech 工业级中文识别，并且做了关键的安全设计——订阅密钥绝不下发前端，后端只签发 10 分钟的短时令牌，浏览器拿令牌直连 Azure。"

可选对比展示：对同一句"下周三下午三点跟客户对接"，展示 Web Speech 识别错 vs Azure 识别对（[docs/SPEECH_CHOICE.md](SPEECH_CHOICE.md) 有论证）。

## 3. 自然语言时间解析（1:30–2:30）

口播 + 现场说这几句（看右侧日历落到正确日期）：
- "**下周三**下午三点开会" → 落到下周三 15:00
- "**每周一三五**早上九点晨会" → 一次展开成多天（指着日历看多个事件）
> "我们实测过 dateparser 对中文几乎没用——'明天下午三点''三天后'全都解析失败。所以中文时间解析是我们**自己写的规则引擎**，覆盖 30 多种表达式，循环日程还能自动展开。"

## 4. 对话闭环 + 澄清 + 冲突（2:30–3:30）

- 说"**把那个会删了**"（库里有多个会）→ 系统语音反问"找到 N 个：…，要删哪一个？"（歧义澄清）
- 说"明天下午三点再加个电话会议"（与已有冲突）→ 系统语音回"这个时间和你的 X 冲突了，要不要改到 16 点？"（冲突建议）
> "注意系统全程**语音回应**，不是默默操作——这就是题目要求的'以语音交互为核心'的对话闭环。"

## 可能被问到的问题（预案）

- **Q：语音指令会传到云端吗？隐私？**
  A：识别走 Azure（企业级、合规），后端只存日程文本不存音频；密钥不下发前端，只发短时令牌。
- **Q：离线场景怎么办？**
  A：识别需联网（工业级准确率的取舍）；可降级到浏览器原生 Web Speech 作保底。日程数据本地 SQLite。
- **Q：时间解析为什么不用现成库？**
  A：实测 dateparser 中文几乎不可用（demo 可现场展示它返回 None），所以自写规则引擎，确定、可测、能展开循环。
- **Q：识别错了怎么办？**
  A：识别中实时回显文字可发现错误；意图模糊时系统主动澄清而非瞎执行；还有文字输入兜底。
- **Q：AI 写了多少？**
  A：Claude Code 辅助开发，但每个关键决策有人工把关、单测验证、真实联调；复盘.md 记了 AI 出错与修正的真实例子。
