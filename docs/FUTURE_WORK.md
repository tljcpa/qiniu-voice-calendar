# 未实现与扩展方向

> 72 小时必有取舍。诚实列出未做的，以及"以后怎么扩展"。

## 已实现（核心闭环完整）

- Azure 工业级中文 ASR + TTS 语音闭环
- 自然中文时间解析（30+ 表达式 + 循环展开）
- 意图解析（加/删/查/改/澄清）+ 实体抽取
- 冲突检测 + 空闲时段建议
- 两级歧义澄清（语言层 + 数据层）
- 日历视图 + 提醒（浏览器通知）
- HTTPS 上线

## 未实现（受时间限制，非技术不可行）

| 项 | 说明 | 为什么没做 |
|---|---|---|
| 多用户 / 账号体系 | 当前单用户单库 | demo 不需要，加 user 维度是过度设计 |
| 农历 / 法定节假日 | "农历八月十五""国庆假期前一天" | 规则难穷举，已留 LLM 兜底层接口 |
| 多轮对话状态 | 澄清后"下午那个"的指代消解 | 需要会话记忆，当前每条指令独立 |
| 离线 / 不在线提醒 | 邮件 / 短信 / 公众号推送 | 浏览器通知够 demo；离线推送需 APScheduler + 渠道接入 |
| 与 Google/Apple 日历同步 | 双向同步 | 需 OAuth 与各家 API 适配 |
| 时区支持 | 当前 naive datetime 单时区 | demo 单时区够用 |

## 下一步规划（评委可能问"以后怎么扩展"）

1. **多轮对话与指代消解**：引入会话状态，记住上一轮候选，支持"改下午那个""就第一个"。当前架构里意图解析已能输出 `target_query`，补一层会话上下文即可。
2. **离线提醒**：把提醒从"浏览器轮询"升级为 APScheduler + sqlite jobstore 服务端定时，接邮件/公众号，实现人不在线也能收到（见 [复盘 D-09](复盘.md) 已说明触发条件）。
3. **时间解析增强**：把 LLM ISO 兜底层接上，覆盖农历、节假日；规则层继续补高频表达。
4. **日历同步**：接 CalDAV / Google Calendar API，做双向同步。
5. **个性化**：学习用户习惯（常用地点、默认时长、常见参会人），加事件时智能预填。

## 设计上为扩展留的余地

- LLM 抽象层双后端 + 自动 fallback：加开源/本地模型只需加一个 base_url 后端。
- 时间解析三层结构：LLM 兜底层接口已留，接上即覆盖异常表达。
- 提醒 channel 字段已存（browser/email…），加渠道不改 schema。
- 后端 ASR/TTS 能力保留 + 预留 `/ws/asr`：需要服务端中转时可切换。
