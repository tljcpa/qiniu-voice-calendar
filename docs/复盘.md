## 1. 关键决策日志（D-XX）

> 本项目必须拍板的关键问题（子 CC 边做边填）：

### D-01: 语音方案（Azure Speech vs Web Speech API vs 阿里云）
- 选项 A：Azure Speech SDK（已开通 F0 免费层 500k 字符/月，中文 zh-CN）
- 选项 B：浏览器原生 Web Speech API（零依赖、免费）
- 选项 C：阿里云 / 七牛云 ASR（需额外开户配置）
- **选 A**。理由：(1) Web Speech API 在中文场景识别率明显偏低，且 Safari、部分国产浏览器（UC/夸克）不支持，移动端不可控；(2) Azure 是工业级方案，中文准确率、流式、抗噪显著更好，且支持 TTS 做"语音闭环"；(3) C 在已有 Azure 配额下属重复投入。
- 为什么不选 B：识别率不能看 + 兼容性差，做不出"准确顺畅"的核心卖点。但仍把它保留为**降级方案**（Azure 不可用时前端回退），代价几乎为零。
- 改变此决策的条件：Azure F0 配额耗尽（500k 字符≈数千次 demo 指令，远超需要）或 eastus2 区域不可用。
- 现状验证：2026-05-29 已用 issueToken 端点确认 key 有效（HTTP 200）。

### D-02: ASR 模式（流式 vs 批式）+ 在浏览器还是后端跑
- 决策：**浏览器端 Azure JS SDK 做流式 ASR**；后端 `speech.py` 保留服务端 ASR 作验证脚本/降级。
- 为什么浏览器端：省掉"音频上传后端再转发 Azure"一跳，延迟更低，能实时回显识别中文字，demo 观感好。
- 为什么不暴露 key：后端 `/api/speech/token` 用 key 换**短时令牌**（~10min）下发前端，前端拿令牌直连 Azure。key 进前端即泄露会被盗刷 F0 配额；令牌失效快且不可反推 key——这是 Azure 官方推荐的浏览器集成方式。
- 改变条件：需要服务端统一审计/录音留存时，改走后端 WS 中转（已预留 `/ws/asr`）。

### D-03: 是否启用 TTS 反馈（真"语音闭环" vs 仅语音输入）
- 决策：**启用 TTS**。前端拿同一短时令牌直接调 Azure TTS 合成并播放系统回应（"已添加6月3日下午三点的产品评审会"）。
- 为什么：题目要求"以语音交互为核心"，只做语音输入是"加个录音框"，加 TTS 才是对话闭环——直接命中评分亮点 2。
- 为什么不后端合成音频再传前端：多一跳延迟且占带宽，前端直连更快；后端 TTS 能力仍保留作降级。
- 权衡：F0 配额 ASR+TTS 共享 500k 字符/月，回应文案短（数十字），消耗可忽略。

### D-04: 中文时间解析方案（dateparser / LLM JSON / 自写规则 / 三层兜底）
- 原计划（BRIEF §5.3）：LLM 抽表达 → dateparser 解析 → 规则补丁兜底（dateparser 为主）。
- **实测推翻该顺序**（见 L-04）：dateparser 1.4 对中文日历表达几乎全废，"明天下午三点""三天后""五月二十号""下周三"全返回 None，只有"周一"勉强且给午夜。
- 修正后决策：**自写确定性规则层为主力**（覆盖相对天/周内某天+周修饰/年月日/时间点/循环），dateparser 仅作边角兜底（标准数字日期），LLM ISO 转换作最后兜底（农历、节假日等异常表达）。
- 为什么规则为主：日历指令的时间表达高度模式化，规则确定、可测、零延迟、零成本，且循环（"每周一三五"）必须确定性展开——LLM 时间算术不可靠。
- 为什么保留 LLM 兜底：覆盖"国庆假期前一天""农历八月十五"这类规则难穷举的表达，标低置信度。
- 改变条件：出现规则覆盖不了的高频表达，优先补规则而非依赖 LLM。

### L-04: dateparser 中文支持几乎不可用
- 现象：dateparser 1.4 + languages=['zh'] 解析"明天下午三点/三天后/五月二十号/下周三/每周一三五"全部返回 None。
- 原因：dateparser 的中文规则只覆盖极少标准格式，相对表达和口语基本不认。
- 解决：不依赖它做主解析，改自写规则层为主。
- 教训：第三方库的"支持某语言"要实测验证覆盖面，别按文档假设；BRIEF 的预设方案也要用证据校验。

### D-05: 意图分类粒度（add / del / view / update / clarify 还是更细）
- 决策：核心 5 类 + 兜底——`add`(添加) / `delete`(删除) / `view`(查询) / `update`(修改) / `clarify`(信息不全需澄清) / `unknown`(无关或无法识别)。
- 为什么这 5 类：直接对应题目"添加/删除/查看事件提醒"三大能力 + 修改（高频）+ 澄清（对话式 UX 亮点 4）。不再细分（如"批量添加"单列）——批量是 add 的多事件形态，由实体里的列表承载，不增意图类别。
- 为什么 LLM 只抽 `time_expr` 原始表达、不做时间归一化：归一化交给 time_parser 三层兜底（D-04），职责分离，LLM 时间算术不可靠。
- `clarify` 由谁触发：LLM 判定关键槽位缺失（如 delete/update 找不到明确目标，或 add 缺时间）时返回 clarify + 要追问的问题 + missing 槽位列表。
- 改变条件：用户反馈某类指令频繁落入 unknown，再加意图类别或补 few-shot。

### D-15: 意图解析 prompt 设计（JSON mode + few-shot + 槽位 schema）
- 决策：system prompt 固定输出 schema，用 `complete_json` 强约束；带 4-6 条覆盖各意图的 few-shot；reminder 等数值槽位缺省为 null 而非猜测。
- 为什么 few-shot：中文口语指令变体多（"删掉""取消""不要了"都是 delete），靠示例让模型稳定归类，比纯指令描述可靠。
- 为什么数值缺省 null：宁可后续追问也不臆测时间/人数，避免"幻觉填充"造成错误日程。
- 测试：单测注入假 LLM 返回固定 JSON，验证解析与校验逻辑；prompt 真实效果用一次性脚本跑多条话术人工调试（本机，非服务）。

### D-14: Azure Speech 后端职责（token 签发 vs 服务端 ASR/TTS）
- 决策：后端 `speech.py` 的**核心职责是签发短时 token**（httpx 调 issueToken REST，带本地缓存，token ~10min 有效），供浏览器 Azure JS SDK 直连用；服务端 TTS 合成作为验证/降级能力（懒加载 azure SDK）；服务端流式 ASR 仅作降级，放验证脚本里在 VM 跑。
- 为什么 token 签发用 httpx 而非 azure SDK：签发就是一个 POST，无需引入重型 SDK；token 端点逻辑可用 mock httpx 完整单测，本机即可验证（遵 D-12）。
- 为什么要缓存 token：F0 有调用频率限制，且每次录音都换 token 浪费；缓存到过期前复用，降低延迟与配额消耗。
- 为什么不把 key 直接给前端：见 D-02，会被盗刷。
- 改变条件：若需服务端统一录音/审计，把 ASR 主路径搬到后端 WS。

### D-13: LLM 抽象层设计（一套客户端封装双后端 + JSON mode）
- 决策：用 openai-python 一个库封装两个后端——DeepSeek（默认，`base_url` 覆盖）与 Azure OpenAI（备用，`AzureOpenAI` 客户端）。统一接口 `complete()` 返回文本、`complete_json()` 返回 dict。
- 为什么：两者都讲 OpenAI 协议，一套调用代码 + 配置切换后端，避免为每家写一套；意图解析/实体抽取强依赖结构化输出，故一等公民支持 `response_format={"type":"json_object"}`。
- 自动 fallback：主后端异常（网络/限流/余额）时自动切备用后端重试一次，保证 demo 不因单点挂。
- 为什么不那么做：不用 LangChain（套壳、调试不可控）；不自己写 HTTP（openai-python 已处理重试/超时/流式）。
- 改变条件：需要本地/开源模型时，因都是 OpenAI 协议，加一个 base_url 后端即可。
- 测试：单测用 monkeypatch 把 client 换成假对象，脱离网络与凭证即可在本机跑（遵 D-12）。

### D-06: 歧义澄清策略（如何检测意图模糊度并主动追问）
- 两级澄清：① LLM 在意图解析阶段判定信息不全直接返回 intent=clarify（缺时间、目标模糊）；② 编排阶段按数据库实际命中数判定——delete/update 时 find_events 命中 0 个→"没找到"，命中 1 个→直接执行，命中 ≥2 个→列候选追问"要哪个"。
- 为什么两级：LLM 只看语言层面（"那个会"显然模糊），但"我的评审会"是否模糊取决于库里有几个——必须查库才知道，这层只能在编排里做。
- 澄清话术带候选列表（带时间），让用户一句话就能选（"下午那个"）。
- 改变条件：引入多轮对话状态后，可记住上一轮候选，支持指代消解。

### D-16: 语音指令编排与冲突处理策略
- 编排链：parse_intent → 按意图分支 → parse_time（add/view 需要）→ CRUD/冲突 → 生成 TTS 回应文案。统一返回 {intent, ok, speech, needs_clarification, clarification, candidates, events}。
- 冲突处理：add 命中冲突时**默认不创建**，返回冲突事件 + 建议时段，speech 提示"和X冲突，要不要改到Y"；带 force=True 时强制创建（用户坚持）。
- 为什么默认不创建：showcase 冲突检测亮点，避免静默制造重叠日程；force 留出"我就要这个时间"的出口。
- 循环事件（每周一三五）：parse_time 展开的多个时间点逐个建，speech 汇总"已添加 N 个"。
- speech 文案是闭环关键（D-03）：每个分支都生成自然中文回应，供前端 TTS 播报。
- 测试：注入假 LLM 返回固定意图 + 真实内存库 + 真实 time_parser，端到端覆盖各分支，脱网。

### D-07: 冲突检测算法（时间区间重叠的判断与建议生成）
- 重叠判定：两区间 [s1,e1)、[s2,e2) 相交 iff `s1 < e2 且 s2 < e1`（半开区间，相邻不算冲突，如 15:00 结束与 15:00 开始不冲突）。
- 建议生成：检测到冲突时，从期望开始时间起以 15 分钟步进向后扫描，找第一个能容纳该时长且不与任何事件冲突的空档，限制在合理工作时段（默认 8:00-22:00）与当天/次日，给出"挪到 X" 的建议。
- 为什么半开区间：日程紧邻是常态（一个会刚结束下一个开始），闭区间会误报。
- 为什么 15 分钟步进：日程通常按刻钟对齐，步进太细无意义、太粗错过空档。
- 为什么不只报错而给建议：体现"日历助手"产品思维（亮点 5），冲突时主动给可行替代。
- 改变条件：需要跨天长周期排程或考虑事件优先级时，升级为更复杂的排程算法。

### D-10: 数据库 schema（事件、提醒、用户设置）
- 决策：两表——event(id/title/start_at/end_at/location/attendees/note/created_at) + reminder(id/event_id/remind_at/channel/sent)。SQLAlchemy 2.0 ORM + SQLite。
- attendees 存 JSON 列（list），不另开关联表：demo 规模下参与人只读不查询，JSON 最简；要按人查日程时再范式化。
- end_at 可空：用户口语很少说结束时间，缺省 = start_at + 1 小时（在 crud 层补，不污染模型）。
- 时间存 naive datetime（本地时区语义）：demo 单用户单时区，不引入 tz 复杂度；上线多时区再加 tz 列。
- 为什么不加 user 表：demo 单用户，加 user 维度是过度设计；预留 event.owner 字段空间但暂不实现。
- 改变条件：多用户/多时区/按参与人检索 → 加 user 表、tz、attendee 关联表。

### D-12: 开发/测试拓扑（本机 vs Azure VM 分工）
- 背景：本机内存仅 3.8G 且已吃 swap，跑 dev server/uvicorn/npm 易 OOM。
- 决策：**本机只做 写代码 / commit / push / 纯单元测试（mock 外部依赖）**；任何起服务、前端 dev、真实 LLM/语音/集成测试一律 push 后 ssh 到 Azure VM（20.172.25.105），代码拉到 `/opt/voice-calendar` 在 VM 上跑。

> 子 CC 边做边填。每个坑都值钱。

### L-01: 本机 Python 是 3.10 而非 BRIEF 推荐的 3.11
- 现象：`python3 --version` → 3.10.12，BRIEF §5.1 推荐 3.11。
- 原因：开发机系统自带版本。
- 解决：开发用 3.10（FastAPI / dateparser / azure-speech 全兼容，不阻塞）；Docker 镜像显式钉 `python:3.11-slim` 保证部署环境与推荐栈一致，避免"本地能跑线上挂"。
- 教训：开发版本和部署版本解耦，用容器锁定部署版本，本地不必强制升级。

### L-02: 开发机未预装 gh CLI
- 现象：`gh: command not found`。
- 解决：下载 gh 2.63.2 预编译二进制到 `~/.local/bin/gh`，用 `GH_TOKEN` 鉴权（repo scope，足够开/合 PR）。
- 教训：每次 Bash 调用前需 `export PATH="/home/claude/.local/bin:$PATH"`（shell 状态不跨调用持久化）。

### L-03: 验证 uvicorn 时端口 8099 已被占用，curl 打到了别的服务
- 现象：起 uvicorn 后 curl /health 返回 `{"detail":"Not Found"}`，但 TestClient 测试却通过；openapi 路由显示是 `/api/health`（不是我的路由）。
- 原因：开发机/本机已有其它服务占用 8099（BRIEF §3.5 警告的"三项目共机"），我的 uvicorn 实际 bind 失败 Exit 1，curl 命中了那个旧服务。
- 解决：换空闲高端口验证通过；部署严格按 BRIEF 用 8081 + 独立 compose project + 独立目录。
- 教训：共享机上验证服务必须先确认端口空闲（`ss -ltnp`），否则"测到的"可能是别人的进程。健康检查返回服务名字段（service=voice-calendar-backend）正是为了一眼区分打到的是不是自己。

### 5.4 AI 出过的错与修正

- **意图解析过度澄清（PR9 实测发现）**：命令行端到端跑真实 LLM 时，"把客户对接删了"被判成 clarify，且 LLM 编造"你有多个客户对接"——可库里只有一个。LLM 在**语言层过度澄清**，还臆测了它根本无法知道的 DB 事实（有几个）。
  - 修正：改 prompt——给出具体名称就走 delete/update（把描述放 target_query），由后端查库判断歧义；只有纯指代（"那个""刚才那个"）才 clarify；明确禁止 LLM 编造"你有多个X"。
  - 验证：复跑后"把客户对接删了"→delete、"把那个删了"→clarify，符合 D-06 两级澄清设计。
  - 教训：LLM 不该替系统编造数据库状态；"是否歧义"在不同层有不同判据（语言层 vs 数据层），职责要分清。

- **dateparser 误信文档（PR6，见 L-04）**：按 BRIEF 假设 dateparser 能解析中文，实测几乎全废，据证据改规则层为主。
